## 1. System Overview

LTRaaS takes a user-defined scan config, generates adversarial probes against a target LLM application, judges each response, and produces a normalised finding report.

```
scan_config.yaml
      │
      ▼
  main.py  ──── validates config ──── builds engine ──── registers target
      │
      ▼
 orchestrator ──── scan_translator ──── [Job, Job, ...]
      │                                        │
      │                               ┌────────┴────────┐
      │                          capability_matrix.yaml  │
      │                               └────────┬────────┘
      ▼                                        ▼
   worker ──── fetches key from vault ──── runs engine
      │
      ▼
  normalizer ──── groups runs ──── assigns severity ──── cluster IDs
      │
      ▼
  reporter ──── console summary ──── outputs/report_<scan_id>.json
```

**Three engines, one pipeline.** Swapping engines (mock → llm → promptfoo) changes only `scan_config.yaml`. All other modules are engine-agnostic.

## 3. Data Models

All dataclasses live in [ltraas/types.py](ltraas/types.py). They are the shared contract between every module — nothing passes raw dicts across module boundaries.

### Target
```
Target
  id            str   "tgt_<8hex>"   — generated by new_id()
  name          str                  — human label, e.g. "PizzaBot Production"
  endpoint_url  str                  — full HTTPS URL the worker POSTs probes to
  system_type   str                  — "chatbot" | "rag" | "tool_agent"
  purpose       str                  — free text; attack generator uses this to craft probes
  auth_ref      str                  — vault key, e.g. "vault://targets/tgt_abc/api_key"
                                       NEVER the raw API key
```

### ScanDefinition
```
ScanDefinition
  id                    str         "scn_<8hex>"
  target_id             str         foreign key → Target
  name                  str
  categories            list[str]   subset of: prompt_injection | secret_leak |
                                                off_topic | jailbreak
  llm_repetitions       int         runs per probe (mock / llm engines)
  promptfoo_num_tests   int         test cases per plugin (promptfoo engine)
  promptfoo_concurrency int         max parallel requests to target (promptfoo engine)
```

### Job
```
Job
  id           str   "job_<8hex>"
  scan_id      str   foreign key → ScanDefinition
  target_id    str   foreign key → Target
  engine       str   "promptfoo" | "llm" | "mock"
  category_id  str   the unified category this job covers
  config_blob  dict  full engine config; contains {{TOKEN}} placeholder (not the real key)
  repetitions  int   how many times to run each probe
```
One `ScanDefinition` produces one `Job` per `category_id`. For promptfoo, the job config is a full promptfoo YAML shape; for llm/mock, it is a minimal target + purpose dict.

### ScanPlan
```
ScanPlan
  scan_id  str
  jobs     list[Job]   all jobs for one scan
```

### RawResult
```
RawResult
  job_id      str
  engine      str           runtime engine name (e.g. "promptfoo", "llm", "mock")
  status      str           "succeeded" | "failed"
  raw_output  list[dict]    verbatim engine events (see normaliser section)
  error       str | None
```

Each event in `raw_output` must have these keys (normaliser contract):
```
  plugin         str   engine plugin ID or unified category (mock/llm use category_id)
  probe          str   the adversarial prompt text
  response       str   target LLM's reply
  graderVerdict  str   "pass" | "fail"
  graderReasoning str
  run            int   run index within repetitions
```

### Finding
```
Finding
  id               str   "fnd_<8hex>"
  scan_id          str
  target_id        str
  category_id      str   unified category (after reverse_lookup for promptfoo)
  severity         str   "low" | "medium" | "high" | "critical"
  engine_source    str   which engine ran this
  probe_text       str   the adversarial prompt
  target_response  str   representative response (first failing run, or first run)
  judge_verdict    str   "pass" | "fail"
  judge_reasoning  str
  runs             int   total runs of this probe
  successes        int   runs where graderVerdict == "fail"
  success_rate     float successes / runs  (0.0 – 1.0)
  plugin_id        str | None   engine-native plugin (e.g. "pii:api-db")
  cluster_id       str | None   dedup key; same vuln across probes shares one ID
  created_at       str   UTC ISO-8601
```

## 4. Module Reference

### main.py — Entry point

**Responsibilities:**
- Load and fully validate `scan_config.yaml` (all errors collected before failing).
- Resolve the target endpoint URL for known providers.
- Build the engine instance.
- Call into `storage`, `target_registry`, `orchestrator`, `reporter` in sequence.

**Key functions:**

| Function | What it does |
|---|---|
| `_validate_config(cfg)` | Collects every missing / invalid field into a list; raises one `ValueError` with all of them at once. Engine-specific fields are only checked for the chosen engine. |
| `load_config()` | Calls `_validate_config`, then for known providers builds `target_model["url"]` from `provider + model` via `build_target_url()`. Returns the complete config dict. |
| `build_engine(cfg)` | Returns `None` (mock), `LLMEngine(...)`, or `PromptfooEngine()` based on `cfg["engine"]`. |
| `main()` | Orchestrates steps 0–4: init DB → register target → define scan → run scan → generate report. |

**Validation matrix:**

| Field | Required when |
|---|---|
| `engine` | Always |
| `models.target.provider` | Always |
| `models.target.model` | provider ∈ {google, openai, anthropic} |
| `models.target.url` | provider == custom |
| `models.target.request_body` | provider == custom |
| `models.target.transform_response` | provider == custom |
| `models.target.api_key_env` | Always |
| API key present in env or config | engine ≠ mock |
| `models.judge.provider` + `.model` | engine ∈ {llm, promptfoo} |
| `models.attack_generator.provider` + `.model` | engine == promptfoo |
| `target.name`, `system_type`, `purpose` | Always |
| `scan.name`, `scan.categories` | Always |
| `scan.llm_repetitions` | engine ∈ {mock, llm} |
| `scan.promptfoo_num_tests` + `promptfoo_concurrency` | engine == promptfoo |

### ltraas/storage.py — Persistence

Three backing stores, all accessed through functions — no direct SQL elsewhere.

**SQLite** (`data/ltraas.db`):
- `init_db()` — creates tables; runs migration to add `plugin_id` column if it predates that field.
- `save_target(t)` / `get_target(target_id)` — INSERT OR REPLACE / SELECT.
- `save_scan(...)` / `update_scan_status(...)` — scan lifecycle.
- `save_finding(f)` / `findings_for_scan(scan_id)` — INSERT / SELECT all for a scan.

**Artifacts** (`data/artifacts/`):
- `save_artifact(scan_id, job_id, data)` — writes `<scan_id>_<job_id>.json`; returns the path.
- Called twice per job: once for the normalised events, once for the raw promptfoo JSON (`_promptfoo_raw` suffix).

**In-memory vault** (`_VAULT: dict`):
- `vault_put(ref, secret)` — stores under `ref`.
- `vault_get(ref)` — retrieves; raises `KeyError` if absent.
- Vault refs follow the pattern `vault://targets/<target_id>/api_key`.

---

### ltraas/target_registry.py — Target management

```python
register_target(name, endpoint_url, api_key, system_type, purpose) -> Target
```
1. Generates `target_id = new_id("tgt")`.
2. Derives `auth_ref = "vault://targets/<target_id>/api_key"`.
3. Calls `vault_put(auth_ref, api_key)` — key never touches the DB.
4. Constructs `Target(...)` (no `api_key` field on the dataclass).
5. Calls `storage.save_target(target)`.
6. Returns `Target`.

```python
get_target(target_id) -> Target      # raises ValueError if not found
get_api_key(target)   -> str         # fetches from vault just-in-time
```

---

### ltraas/orchestrator.py — Scan runner

```python
submit(scan_def, engine_impl, models, engine_type) -> str
```
Sequential pipeline:
1. Looks up the `Target` from the registry.
2. Persists the scan record with `status="running"`.
3. Calls `scan_translator.build_plan()` → `ScanPlan`.
4. For each `Job` in the plan:
   - Calls `worker.process_job(job, engine_impl)` → `RawResult`.
   - Calls `normalizer.normalize(job, raw)` → `list[Finding]`.
5. Updates scan status to `"complete"`.
6. Returns `scan_def.id`.

All errors in a job are logged but do not abort the scan — remaining jobs still run. A failed job produces an empty `RawResult` with `status="failed"` and yields zero findings.

### ltraas/scan_translator.py — Job builder

**Public API:**

```python
build_target_url(provider, model) -> str          # "google" → full Gemini URL
reverse_lookup(engine, engine_probe_id) -> str    # "pii:api-db" → "secret_leak"
build_plan(scan_def, target, models, engine_type) -> ScanPlan
```

**`build_plan` routing:**

```
engine_type == "promptfoo"  →  _build_plan_promptfoo()
engine_type ∈ {"llm","mock"} →  _build_plan_direct()
```

**`_build_plan_promptfoo`** — consults the capability matrix:
- Skips categories not in the matrix (warns).
- Skips categories where `target.system_type` is not in `applies_to` (info log).
- Calls `_render_promptfoo_config()` to build the full promptfoo YAML shape.

**`_build_plan_direct`** — does NOT consult the matrix:
- One job per requested category, regardless of `system_type`.
- Config blob contains only `targets` + `redteam.purpose`.

**`_render_promptfoo_config`** produces:
```
{
  targets: [{id: "http", label: <name>, config: <target_cfg>}],
  evaluateOptions: {maxConcurrency: <n>},
  defaultTest: {assert: [{type: "llm-rubric", provider: <judge_id>, value: <rubric>}]},
  redteam: {
    purpose: <purpose>,
    plugins: [{id: p} for p in entry["plugins"]],
    strategies: [{id: s} for s in entry["strategies"]],
    numTests: <promptfoo_num_tests>,
    provider: <attack_generator_id>,
  }
}
```
`{{TOKEN}}` appears in `target_cfg` but is not substituted here — the worker does it at runtime.

**`_PROVIDER_CONFIG`** — routing table for known providers:

| Provider | URL template | Key placement | Response path | promptfoo prefix |
|---|---|---|---|---|
| google | `generativelanguage.googleapis.com/…/{model}:generateContent` | `?key={{TOKEN}}` in URL | `json.candidates?.[0]?.content?.parts?.[0]?.text` | `google` |
| openai | `api.openai.com/v1/chat/completions` | `Authorization: Bearer {{TOKEN}}` | `json.choices?.[0]?.message?.content` | `openai` |
| anthropic | `api.anthropic.com/v1/messages` | `x-api-key: {{TOKEN}}` | `json.content?.[0]?.text` | `anthropic` |

**Custom provider** (`_build_custom_target_config`):
- Uses `t_models["request_body"]` as the body template verbatim.
- Uses `t_models["transform_response"]` as the JS expression.
- Optionally adds an auth header if `auth_header` is set.

### ltraas/worker.py — Engine executor

```python
process_job(job, engine_impl=None) -> RawResult
```
Lifecycle:
1. Resolves engine (defaults to `MockEngine` if `engine_impl` is `None`).
2. Fetches `api_key` from vault via `target_registry.get_api_key(target)`.
3. Calls `_substitute(job.config_blob, {"TOKEN": api_key})` — walks the JSON tree and replaces `{{TOKEN}}` with the real key.
4. Calls `engine.run(job, config_with_secret)`.
5. In `finally`: zeroes `api_key` and `config_with_secret`.
6. Saves raw output to artifact storage.
7. Returns `RawResult`.

**`_substitute(obj, mapping)`** — serialises the dict to JSON, replaces all `{{KEY}}` occurrences with the mapping value, then deserialises. Simple and safe for string substitution in nested dicts.

#### MockEngine
- No network or API keys.
- Picks probes from `PROBE_LIBRARY[category_id]`.
- Simulates a pass/fail verdict using per-category vulnerability rates:

| Category | Simulated break rate |
|---|---|
| prompt_injection | 25% |
| secret_leak | 40% |
| off_topic | 55% |
| jailbreak | 15% |

- Uses `random.seed(42)` set in `main()` for reproducible results.
- Returns results in the same shape as `PromptfooEngine` so the normaliser is identical.

#### LLMEngine
- Sends real HTTP probes directly to the target using `urllib.request`.
- Judges each response using a separate judge LLM.
- Supports google, openai, and anthropic as both target and judge providers.
- Infers the target provider from the URL via `_detect_provider(url)` for response parsing.
- `_CATEGORY_CRITERIA` maps each category to a judge instruction string.
- Handles timeouts per probe (logs a warning, marks as pass/inconclusive, continues).
- Judge response: expects `{"verdict": "pass"|"fail", "reasoning": "..."}`. Falls back gracefully on malformed JSON or empty responses.

```
_call_target(probe, url, headers, body_template) → str
  POST probe → target URL
  parse response via _extract_response(data, provider)

_judge(probe, response, purpose, criteria) → (verdict, reasoning)
  build judge request via _build_judge_request(system, user)
  POST to judge API
  parse JSON verdict
```

#### PromptfooEngine
- Shells out to the `promptfoo` CLI (`npm install -g promptfoo` required).
- Writes the config blob to a temp file, runs `promptfoo redteam run`, reads `results.json`.
- Auto-installs promptfoo via `npm install -g promptfoo` if not found on PATH.
- Exit code `100` = some tests failed — expected; not treated as an error.
- Saves full raw JSON before the temp dir is deleted.
- `_normalize(data, plugin)` maps promptfoo's result shape to the shared event dict format:
  - Extracts `pluginId` from `r.metadata.pluginId` or `r.testCase.metadata.pluginId`.
  - Finds the first failing component's reason (avoids contradictory "safe" messages from passing sub-graders).
- Reads API keys from environment variables directly (not from the vault).

### ltraas/normalizer.py — Finding builder

```python
normalize(job, raw) -> list[Finding]
```

Steps:
1. Returns `[]` immediately if `raw.status != "succeeded"`.
2. Groups events by `event["probe"]` — all runs of the same probe text go together.
3. For each probe group:
   - `successes = count(graderVerdict == "fail")`.
   - `success_rate = successes / len(runs)`.
   - **Category resolution**: promptfoo → `reverse_lookup(engine, plugin_id)` → unified category; llm/mock → `job.category_id` directly.
   - **Representative event**: first failing run, or `runs[0]` if all passed.
   - Builds `Finding`, computes `cluster_id`, saves to DB.
4. Returns the full list.

**`_severity_for(category_id, success_rate)`:**

Base score by category:

| Category | Base score |
|---|---|
| off_topic | 2 |
| prompt_injection | 3 |
| secret_leak | 4 |
| jailbreak | 4 |
| unknown | 2 |

Rate bump added to base score:

| success_rate | Bump |
|---|---|
| 0 | –∞ (overrides to "low") |
| < 10% | +0 |
| 10–49% | +1 |
| ≥ 50% | +2 |

Final score → severity: `2=low`, `3=medium`, `4=high`, `≥5=critical`.

**`_cluster_id(finding)`:**  
`MD5(target_id + "|" + category_id + "|" + first_5_words_of_probe)[:12]`  
Same probe phrasing hitting the same category on the same target → same cluster ID across multiple scans.

---

### ltraas/reporter.py — Report generator

```python
generate(scan_id) -> dict
```
Reads all `Finding` rows for the scan and produces:

**Console output** (logged at INFO — appears on both console and log file):
- Header with scan ID.
- Total probes / broke / unique vulnerabilities / break rate.
- Per-category breakdown: `{breaks}/{total} broke ({pct}%)`.
- Per-plugin breakdown (if any plugins present).
- Per-severity counts (critical → high → medium → low).
- Top 3 failing probes with probe text, response, and judge reasoning (truncated to 80 chars).

**JSON file** at `outputs/report_<scan_id>.json`:
```json
{
  "scan_id": "scn_...",
  "summary": {
    "total_probes": 10,
    "broken": 4,
    "unique_vulnerabilities": 3,
    "by_category": {"secret_leak": 5, "off_topic": 5},
    "by_plugin": {"pii:api-db": 3, "rbac": 2},
    "by_severity": {"high": 2, "medium": 2}
  },
  "findings": [ ...Finding.to_dict() for each finding... ]
}
```

## 5. End-to-End Data Flow

```
User
 │
 │  python main.py
 ▼
main.py
 ├─ load_config()            reads scan_config.yaml, validates all fields
 ├─ storage.init_db()        creates tables / runs migrations
 ├─ build_engine(cfg)        instantiates MockEngine / LLMEngine / PromptfooEngine
 ├─ target_registry
 │    .register_target(...)  → vault_put(auth_ref, api_key)
 │                             storage.save_target(Target)
 │                             returns Target (no api_key field)
 │
 └─ orchestrator.submit(scan_def, engine_impl, models, engine_type)
      │
      ├─ target_registry.get_target(scan_def.target_id)  → Target
      ├─ storage.save_scan(..., status="running")
      │
      ├─ scan_translator.build_plan(scan_def, target, models, engine_type)
      │    │
      │    │  (for promptfoo engine)
      │    ├─ reads CAPABILITY_MATRIX for each category
      │    ├─ filters by applies_to system_type
      │    ├─ _render_promptfoo_config() → config_blob with {{TOKEN}}
      │    └─ returns ScanPlan([Job, Job, ...])
      │
      └─ for each Job:
           │
           ├─ worker.process_job(job, engine_impl)
           │    │
           │    ├─ target_registry.get_api_key(target)     → api_key from vault
           │    ├─ _substitute(config_blob, {TOKEN: api_key}) → resolved config
           │    │
           │    ├─ engine.run(job, config_with_secret)
           │    │    │
           │    │    │  (PromptfooEngine)
           │    │    ├─ writes config to tempfile
           │    │    ├─ subprocess: promptfoo redteam run --config ... --output ...
           │    │    └─ _normalize(results.json) → list[event_dict]
           │    │
           │    │    (LLMEngine)
           │    │    ├─ for probe in PROBE_LIBRARY[category]:
           │    │    │    for run in range(repetitions):
           │    │    │      _call_target(probe, url, headers, body_template)
           │    │    │      _judge(probe, response, purpose, criteria)
           │    │    └─ returns list[event_dict]
           │    │
           │    │    (MockEngine)
           │    │    └─ generates fake results from PROBE_LIBRARY
           │    │
           │    ├─ finally: api_key = None, config_with_secret = None
           │    ├─ storage.save_artifact(scan_id, job_id, raw_output)
           │    └─ returns RawResult
           │
           └─ normalizer.normalize(job, raw)
                │
                ├─ group events by probe text
                ├─ for each probe group:
                │    count successes / compute success_rate
                │    resolve category_id (reverse_lookup for promptfoo)
                │    compute severity (_severity_for)
                │    build Finding
                │    compute cluster_id (_cluster_id)
                │    storage.save_finding(finding)
                └─ returns list[Finding]

      storage.update_scan_status(scan_def.id, "complete")

reporter.generate(scan_id)
  ├─ storage.findings_for_scan(scan_id)
  ├─ print console summary
  └─ write outputs/report_<scan_id>.json
```

## 6. Engine Deep Dive

### Engine selection and capability

| Feature | MockEngine | LLMEngine | PromptfooEngine |
|---|---|---|---|
| API key needed | No | Yes | Yes |
| Network calls | No | Yes (target + judge) | Yes (via promptfoo) |
| Node.js needed | No | No | Yes (≥ 18) |
| Probe source | Built-in PROBE_LIBRARY | Built-in PROBE_LIBRARY | AI-generated (attack_generator) |
| Judge | Simulated pass/fail | Configurable LLM judge | llm-rubric + plugin graders |
| Category support | All 4 categories | All 4 categories | capability_matrix entries only |
| system_type filtering | No | No | Yes (applies_to in matrix) |
| Results reproducible | Yes (seed=42) | No | No |

### How categories map to engines

```
User config              capability_matrix.yaml          Engine plugin
─────────────────────────────────────────────────────────────────────
prompt_injection  ──→   plugins: [hijacking]        ──→  promptfoo hijacking plugin
                         strategies: [prompt-injection]

secret_leak       ──→   plugins: [pii:api-db,       ──→  promptfoo PII plugins
                                   pii:session]
                         strategies: [basic]

off_topic         ──→   plugins: [rbac]             ──→  promptfoo RBAC plugin
                         strategies: [basic]

jailbreak         ──→   plugins: [jailbreak]        ──→  promptfoo jailbreak plugin
                         strategies: [basic]
```

For llm and mock engines, the matrix is not consulted — categories are passed directly.

### Probe flow per engine

```
MockEngine:
  PROBE_LIBRARY[category] → [probe1, probe2, probe3]
  each probe × repetitions runs → random broke/pass verdict

LLMEngine:
  PROBE_LIBRARY[category] → [probe1, probe2, probe3]
  each probe × repetitions:
    POST probe → target URL → parse response
    POST (probe, response) → judge URL → parse {verdict, reasoning}

PromptfooEngine:
  attack_generator LLM → AI-crafted probes (numTests per plugin)
  promptfoo runs probes → target URL
  llm-rubric judge + plugin-native graders → verdict per probe
```

## 7. Secrets & API Key Flow

API keys are never stored in plaintext in the database or on disk.

```
scan_config.yaml
  models.target.api_key_env: GEMINI_API_KEY
                                  │
                                  ▼
main.py: api_key = os.environ.get("GEMINI_API_KEY")
                                  │
                                  ▼
target_registry.register_target(api_key=api_key)
  vault_put("vault://targets/tgt_xxx/api_key", api_key)
  save_target(Target)   ← auth_ref stored, NOT api_key
                                  │
                                  ▼
worker.process_job(job)
  api_key = vault_get("vault://targets/tgt_xxx/api_key")
  config_with_secret = _substitute(config_blob, {TOKEN: api_key})
  engine.run(job, config_with_secret)           ← key in memory only
  finally:
    api_key = None
    config_with_secret = None                   ← key cleared immediately
```

**`{{TOKEN}}` placeholder lifecycle:**

| Stage | State |
|---|---|
| scan_translator builds config_blob | `{{TOKEN}}` literal string in config |
| config_blob stored in Job (in memory) | `{{TOKEN}}` literal string |
| config_blob saved as artifact (disk) | `{{TOKEN}}` literal string — never the real key |
| worker calls `_substitute()` | real key substituted in memory only |
| engine runs | resolved config passed in memory |
| finally block | reference zeroed |

PromptfooEngine reads API keys directly from environment variables — promptfoo handles that internally and does not go through the vault or `{{TOKEN}}`.

## 8. Capability Matrix

File: [capability_matrix.yaml](capability_matrix.yaml)

Controls which promptfoo plugins and strategies run for each unified category. Editing this file is the only change needed to add, remove, or remap categories.

**Schema:**
```yaml
<category_id>:
  engine:      "promptfoo"          # which engine handles this category
  plugins:     [list of plugin IDs] # promptfoo plugin IDs
  strategies:  [list of strategy IDs]
  applies_to:  [chatbot | rag | tool_agent]
```

**`applies_to` filtering** — the scan translator skips a category if the target's `system_type` is not in `applies_to`. Example: `secret_leak` does not apply to `tool_agent` targets.

**`reverse_lookup(engine, engine_probe_id)`** — used by the normaliser to map a promptfoo plugin ID back to a unified category. Returns `"unknown"` if not found.

Current matrix:

| Category | Plugins | Strategies | Applies to |
|---|---|---|---|
| prompt_injection | hijacking | prompt-injection | chatbot, rag, tool_agent |
| secret_leak | pii:api-db, pii:session | basic | chatbot, rag |
| off_topic | rbac | basic | chatbot, tool_agent |
| jailbreak | jailbreak | basic | chatbot, rag, tool_agent |

## 11. Provider Routing

### Known providers — endpoint construction

| Provider | URL template | Key method | Body format |
|---|---|---|---|
| google | `https://generativelanguage.googleapis.com/v1beta/models/{model}:generateContent` | `?key=` appended to URL | `{contents:[{role:user,parts:[{text:{{prompt}}}]}]}` |
| openai | `https://api.openai.com/v1/chat/completions` | `Authorization: Bearer {{TOKEN}}` | `{model:..., messages:[{role:user,content:{{prompt}}}]}` |
| anthropic | `https://api.anthropic.com/v1/messages` | `x-api-key: {{TOKEN}}` + `anthropic-version` header | `{model:..., max_tokens:1024, messages:[...]}` |

For **google**, the model name is encoded in the URL path, not in the request body.  
For **openai** and **anthropic**, the `model` field is omitted from the body when empty (valid for proxies).

### LLMEngine — judge provider API

| Provider | URL | Auth header | Key env var |
|---|---|---|---|
| google | `generativelanguage.googleapis.com/…/{model}:generateContent` | `x-goog-api-key` | `GEMINI_API_KEY` |
| openai | `api.openai.com/v1/chat/completions` | `Authorization: Bearer` | `OPENAI_API_KEY` |
| anthropic | `api.anthropic.com/v1/messages` | `x-api-key` + `anthropic-version` | `ANTHROPIC_API_KEY` |

## 12. Error Handling Strategy

| Layer | Failure mode | Behaviour |
|---|---|---|
| `main._validate_config` | Missing / invalid field | Collects ALL errors, raises one `ValueError` listing everything wrong |
| `orchestrator.submit` | Missing `models` or `engine_type` arg | Raises `ValueError` immediately |
| `worker.process_job` | Engine throws any exception | Caught; `RawResult(status="failed", error=str(e))` returned; scan continues |
| `LLMEngine.run` | Single probe times out (`socket.timeout`) | Logged as warning; verdict recorded as "pass / inconclusive"; next probe continues |
| `LLMEngine._call_target` | HTTP error from target | Raises `RuntimeError` with status code and body snippet |
| `LLMEngine._judge` | Empty or malformed judge JSON | Returns `("fail", "Judge returned malformed response: …")` |
| `PromptfooEngine.run` | Exit code not in {0, 100} | Raises `RuntimeError` with stdout tail + stderr snippet |
| `PromptfooEngine._ensure_installed` | `npm install` fails | Raises `RuntimeError` with npm stderr |
| `normalizer.normalize` | `raw.status != "succeeded"` | Returns `[]` immediately; logs nothing extra (worker already logged the error) |
| `target_registry.get_target` | ID not in DB | Raises `ValueError("Target {id} not found")` |
| `storage.vault_get` | Ref not in vault | Raises `KeyError` (Python built-in) |
